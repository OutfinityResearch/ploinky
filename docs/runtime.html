Execution Model

Ploinky Documentation

The runtime layer is responsible for workspace discovery, repository-backed agent discovery, per-agent workspace state, execution-backend selection, and the wave-based startup sequence that feeds the router’s route table.

Workspace State Under .ploinky/

Path Purpose
.ploinky/agents.json Enabled-agent registry plus workspace configuration under the reserved _config key.
.ploinky/repos/ Cloned agent and skills repositories managed by add repo and enable repo.
.ploinky/agents/ Isolated per-agent working directories used by the default run mode.
.ploinky/code/ and .ploinky/skills/ Symlink layer exposing each enabled agent’s source tree and optional skills folder.
.ploinky/deps/ Runtime-keyed global and per-agent Node dependency caches.
.ploinky/keys/ Router and agent Ed25519 key material for secure delegated calls.
.ploinky/logs/ Router, watchdog, and command logs used by status surfaces and debugging workflows.
.ploinky/transcripts/ Encrypted WebChat transcript store with retention and turn-level feedback metadata.

Agent Enablement and Run Modes

enable agent resolves a manifest through repository discovery, enforces alias uniqueness, chooses an auth mode, and stores a normalized record in .ploinky/agents.json. If the static agent passed to start is not already enabled, the CLI enables it automatically before booting the workspace.

isolated

The default mode creates a working directory under .ploinky/agents/<agent>/. The path must stay inside the workspace root.

global and devel

global points the agent at the workspace root. devel points the agent at a repository checkout under .ploinky/repos/<repo>/, enabling live development against a cloned repository.

Aliases become route keys and container names. This is how Ploinky supports multiple instances of the same agent in one workspace without overwriting the registry entry for the canonical instance.

Runtime Backends and Mount Policy

The container path is the default backend and auto-detects podman before docker. Agents may also run through sandbox backends such as bwrap or seatbelt. The actual runtime is recorded in the registry, because sandbox attempts may fall back to a container runtime during service startup.

Profile handling is merged from manifest profiles.default plus the active profile from .ploinky/profile. In dev, code and skills mounts default to read-write. In qa and prod, they default to read-only unless the profile explicitly overrides them.

Manifest volumes are resolved relative to the workspace root unless the manifest provides an absolute host path. Runtime resource planning can also create persistent storage directories under .ploinky/data/ and template environment values from workspace roots, storage paths, secrets, and variables.

Dependency Waves and Readiness Gating

The static agent is expanded into a dependency graph through manifest enable directives. The graph is topologically grouped into waves. Each wave is started, routed, and then blocked on readiness before the next wave is allowed to boot. This guarantees that a dependent agent does not start before its prerequisites are reachable.

Readiness protocol defaults are derived from the manifest: start-only agents default to TCP readiness, while agents exposing an MCP server default to an MCP handshake. Manifests may override the protocol explicitly through readiness.protocol.

Static Agent Responsibilities

The static agent determines the router port and the static asset host path recorded in .ploinky/routing.json. Requests for /, /index.html, and the static agent alias resolve against that host root, while the router also keeps serving first-party management surfaces under their fixed prefixes.

If the static site is requested before the agent is ready, the router serves a short bootstrap page that retries automatically rather than exposing a broken asset path.
