Router Surfaces

Ploinky Documentation

The interface layer combines a supervised local router, prefixed browser surfaces, HTTP and MCP proxy routes, and capability-aware authentication that can operate with tokens, local credentials, or an SSO provider bound through the workspace capability registry.

RoutingServer and Watchdog

cli/server/RoutingServer.js owns HTTP request dispatch. cli/server/Watchdog.js owns process supervision, health-check restarts, restart backoff, and container monitoring. The router exposes a health endpoint at /health, browser assets, blob upload and download endpoints, and both router-level and agent-level MCP routes.

The router uses the active route table from .ploinky/routing.json. Its protected paths are authenticated before dispatch, except where a route is explicitly defined as public or where agent-delegated MCP verification must inspect the request body before making an auth decision.

Browser Surfaces

Prefix Current Behavior
/webtty Browser terminal backed by node-pty, guarded by a per-surface token or an authenticated user session, with global and per-session TTY limits.
/webchat TTY-backed chat surface that persists encrypted transcripts, supports uploads, and can stream speech-to-text or text-to-speech through browser or server strategies.
/webmeet Meeting UI with participant presence, SSE fan-out, optional moderator-agent calls, and identity headers derived from the authenticated user.
/dashboard Management surface with transcript and feedback access controls. The active fallback assets live under cli/server/dashboard/.
/status Read-only status UI that shells out to ploinky status, adds router-side server and workspace-agent summaries, and accepts a dashboard invitation link or WEBDASHBOARD_TOKEN for token access.

The root-level dashboard/ directory is not the active fallback for routed /dashboard. The current implementation resolves fallback dashboard assets from cli/server/dashboard/, which is why the product documentation follows that path.

Authentication Modes

Ploinky supports three practical access patterns. /webtty, /webchat, and /webmeet use the normal router login flow. /dashboard still supports WEBDASHBOARD_TOKEN, and the read-only /status surface reuses the dashboard token flow. Local auth stores hashed users inside a workspace-managed variable such as PLOINKY_AUTH_<ROUTE>_USERS. SSO binds the workspace alias sso to an installed agent that provides the auth-provider/v1 contract.

The auth bridge normalizes authenticated users onto req.user and tracks session cookies for both local and external sign-in paths. Local sessions are handled by the router’s session store, and SSO sessions inherit user identity and tokens from the provider flow.

Capabilities and Secure Delegation

Ploinky reads each installed manifest’s provides and requires blocks into a workspace capability index. Workspace-level bindings, including the sso binding, live alongside other workspace configuration rather than in a separate external registry.

For delegated tool calls, the router signs short-lived invocation tokens with its Ed25519 keypair, forwards a user-context token when needed, and verifies agent-signed caller assertions against the registered agent public keys. Replay caches are applied on both the router-signed and delegated-agent verification paths to narrow the replay window.

MCP and HTTP Service Paths

The router aggregates tools and resources at /mcp, while agent-specific MCP endpoints are exposed under /mcps/<agent>/mcp and /mcp/<agent>/mcp. Agent task-status passthrough lives at /mcps/<agent>/task. Public and authenticated HTTP service routes are defined separately and may inject compact auth headers for downstream agents.

This split lets Ploinky expose first-party routing behavior and agent-specific protocol proxies without forcing every browser surface through the same transport semantics.
