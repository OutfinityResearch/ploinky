Operator Workflow

Ploinky Documentation

Operations in Ploinky center on secret resolution, dependency-cache hygiene, observability, transcript handling, test execution, and the maintenance tooling that keeps browser docs, specs, and command-suggestion context aligned with the code.

Secrets and Variable Resolution

Workspace-managed secrets live in .ploinky/.secrets. The secret injector also walks upward to find the nearest .env file and prefers process environment variables over both file-based sources. This precedence model is shared by auth config, runtime resource templating, dependency tooling, and the LLM helper.

var, vars, echo, and expose are the operator-facing commands for that layer. Variable exposure supports wildcard expansion, but the all-match * pattern deliberately excludes names containing API_KEY so sensitive values are never exposed accidentally through a broad manifest pattern.

Dependency Cache Maintenance

ploinky deps prepare, deps status, and deps clean manage runtime-keyed Node caches under .ploinky/deps/. Global caches are prepared from globalDeps/package.json. Per-agent caches merge that baseline with an agent-local package.json when one exists. A cache is valid only when the runtime key, merged-package hash, and stamp version all match the current inputs.

Container runtimes prepare caches inside a short-lived install container for the target image. Sandbox runtimes prepare caches on the host, but only when the runtime key matches the host runtime family and platform.

Observability and Transcript Retention

The watchdog and router write structured logs under .ploinky/logs/. The CLI currently exposes router log tailing and last-line inspection through logs tail and logs last. The status surface also shells out to ploinky status so operators can compare the CLI view with router-side session and service state.

WebChat transcripts are stored under .ploinky/transcripts/ with per-conversation data-encryption keys wrapped by a master key derived from PLOINKY_TRANSCRIPTS_MASTER_KEY. If no master key exists, the runtime generates one and persists it. Transcript viewer access is gated separately from general dashboard access and can be restricted to SSO roles or allowed for local sessions through explicit environment flags.

Skills and LLM Assistance

default-skills installs every skill directory from a skills repository into the workspace-local discovery roots for Claude-style and generic agent tooling. The copied skill trees are added to .gitignore through a managed marker block so the workspace does not accidentally commit generated local installations.

ploinky-shell and invalid-command fallback logic rely on Achilles LLM tooling. The suggestion prompt reads docs/ploinky-overview.md, inspects configured API-key sources, and uses the current model settings from workspace variables such as ACHILLES_MODEL_PLAN. If CLI behavior changes and that overview file is not updated, the suggestion layer becomes stale even when the main docs are correct.

Testing and Documentation Verification

tests/test_all.sh is the full-stage regression harness. It can optionally test another branch by creating a git worktree when PLOINKY_BRANCH is set. tests/runFailingFast.sh rebuilds one fresh workspace and reruns only the previously failing checks to shorten iteration time after a broken run.

Documentation verification is part of the repository contract. After documentation changes, regenerate docs/specs/matrix.md, copy docs/specsLoader.html from the GAMP asset, verify local links, and run the static-site verifier against the finished docs/ directory.

Use fileSizesCheck.sh when documentation or generated content grows substantially. The canonical script lives at the repository root and is referenced from the coding-style specification.
